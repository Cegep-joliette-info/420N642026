# Git

Votre repo doit contenir un README.md, un INSTALL.md (peut être vide au début) et un fichier ou dossier contenant vos normes.

Pour le projet vous devez utiliser les branches avec les pull requests. Vous pouvez utiliser Git en ligne de commande, dans votre IDE ou avec une application comme Kraken. Important, n'utilisez pas l'AI pour faire des pushs, je ne veux pas voir de push fait par Claude, je dois voir vos noms sur les commits.

Si vous avez besoin de rappels sur Git, vous pouvez consulter ce site: https://learngitbranching.js.org/?locale=fr_FR

Toutes les fonctionnalités deront être réalisés dans des branches séparées. Définissez votre norme pour le nom des branches, par exemple en préfixant le type de branche `feature/nom-de-la-fonctionnalite`, `bugfix/description-du-bug` ou en inscrivant le numéro de la tâche `001-nom-de-la-fonctionnalite` ou simplement avec le nom de la fonctionnalité `nom-de-la-fonctionnalite` ou autre, tant que le nom est clair et que vous êtes cohérents dans votre projet.

Pour les pull requests, un autre membre de l'équipe doit l'accepter, vous ne pouvez pas accepter vos propres pull requests. Attention à laisser la chance à tous d'en faire, établissez votre norme, mais ce que je vous conseil, c'est de définir un "reviewer" pour chaque membre de l'équipe. Encore mieux, changez de "reviewer" à chaque sprint, ça vous permettra de voir en détail ce que les autres font. Par exemple, au sprint 1, tous les pull requests de Philippe sont acceptés par Michel, au sprint 2, tous les pull requests de Philippe sont acceptés par Patrick, etc. Attention, si vous n'acceptez pas les pull requests dans un délais raisonnable (surtout durant les périodes de classe), je pourrais donner l'autorisation à un autre membre de l'équipe d'accepter les pull requests à votre place ce qui vous donnerais zéro pour cette tâche.
